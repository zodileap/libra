# Desktop Frontend Rules

## 组件与布局

- 优先使用 `aries_react` 组件，不要默认用原生 `div` 拼布局。
- `main` 区域页面标题必须挂到 `#desk-app-header-slot`，不要在 `main` 内再造独立 header。
- 多级导航优先使用 `AriMenu`，不要自造树形菜单。

## 文案与交互

- 用户可见文本必须接入 i18n。
- 禁止使用 `window.alert`；错误提示与确认交互统一走 `AriModal`、`AriMessage` 等组件。
- 前端错误文案必须做用户友好化映射，不要直接暴露后端原始错误。

## 样式与主题

- 禁止硬编码颜色、间距、尺寸、圆角、阴影和透明度，优先使用存在的 `--z-*` 变量。
- 使用变量前要先确认变量真实存在；不存在时先补设计系统变量或改用已存在变量。

## 分层约束

- 通用组件层只保留展示骨架与通用交互，不写业务判断。
- 页面、服务、共享层职责要清晰，避免互相反向依赖。
- 不要无意义增加抽象层；能复用现有层级就不要再造一层。

## 测试

- `apps/desktop/tests/ui/playwright` 负责 UI 验证。
- `apps/desktop/tests/e2e/tauri-driver` 负责真实 Tauri 容器端到端验证。
- 新结构需要同时给出对应测试落点，不要只写实现不写验证边界。
